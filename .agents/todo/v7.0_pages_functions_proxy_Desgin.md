# X-Flow v7.0 — Pages Functions 代理独立站详细设计（Desgin）

> **版本代号：** v7.0 Pages Proxy Web Runtime  
> **目标：** 在保留 Tampermonkey 用户脚本的同时，新增可部署到 Cloudflare Pages 的独立站，并复用同一套 UI 体验。

---

## 1. 背景与目标

当前项目运行时为 Userscript，依赖 `GM_xmlhttpRequest` 和 `document-start` 接管机制。v7.0 目标是扩展为“双运行时”：

- **Runtime A（现有）**：Tampermonkey Userscript
- **Runtime B（新增）**：Cloudflare Pages 独立站（Browser + Pages Functions）

### 核心目标

1. 复用已有 UI 组件与样式（`Layout`/`Components`/`TikTokMode`/`player.css`）
2. 使用 Pages Functions 代理解决跨域 API 访问
3. 构建流程同时输出 userscript 与 web 静态站点
4. 给出可直接落地的 Cloudflare Pages 部署配置

### 非目标（v7.0 不做）

- 不重做 UI 为框架化（仍保持 Vanilla DOM）
- 不在 v7.0 内实现 v6 未完成的“视频推荐 UI 闭环”功能（先冻结）
- 不替换现有 Worker 分析链路

---

## 2. 总体架构

```text
src/
  core/ (复用域：UI + player + 样式 + 通用状态)
  runtime/
    adapters/
      userscriptAdapter.ts
      webAdapter.ts
    bootstrap/
      userscriptEntry.ts
      webEntry.ts
functions/
  api/media.ts         # 代理源站 /api/media
  api/recommend.ts     # 可选：代理推荐接口
  api/telemetry/*.ts   # 可选：透传到 worker
```

### 架构原则

- **UI Core 无运行时分支**：不直接引用 `GM_*`、不感知 Pages Functions
- **运行时差异全部下沉到 Adapter**
- **启动策略分离**：userscript 保留接管逻辑，web 仅挂载 app shell

## 3. UI 复用设计

## 3.1 复用边界

直接复用：

- `src/ui/Layout.ts`

- `src/styles/player.css`

改造后复用：

- `src/telemetry/EventCollector.ts`（改为依赖 `StorageAdapter` + `HttpAdapter`）

运行时专属：

- `src/ui/Sandbox.ts`（拆分 userscript / web bootstrap 行为）

## 3.2 运行时抽象接口

定义统一适配层：

```ts
export interface RuntimeAdapter {
  http: {
    getJson<T>(url: string, init?: RequestInit): Promise<T>;
    postJson<T>(url: string, body: unknown, init?: RequestInit): Promise<T>;
  };
  storage: {
    get<T>(key: string, fallback: T): Promise<T>;
    set<T>(key: string, value: T): Promise<void>;
  };
  env: {
    mode: 'userscript' | 'web';
    apiBase: string;
  };
}
```

实现约束：

- `userscriptAdapter`: `GM_xmlhttpRequest` + `GM_getValue/GM_setValue`
- `webAdapter`: `fetch` + `localStorage`（或 IndexedDB，先 localStorage）

## 4. API 与跨域方案（Pages Functions 代理）

## 4.1 问题

浏览器端直接请求源站 API 会受 CORS 限制；userscript 可绕过，web 不可绕过。

## 4.2 方案

前端统一请求同域 `/api/*`；Pages Functions 在服务端转发上游 API：

- `/api/media` -> `${UPSTREAM_ORIGIN}/api/media`
- `/api/recommend` -> `${RECOMMEND_ORIGIN}/api/recommend`（可选）
- `/api/telemetry/*` -> `${TELEMETRY_ORIGIN}/api/telemetry/*`（可选）

## 4.3 代理规则

1. 仅允许白名单 path（防开放代理）
2. 保留必要 query（`isAnimeOnly` 等）
3. 默认透传状态码与 JSON body
4. 设置超时与错误兜底：
   - 上游失败返回标准错误结构 `{ ok:false, code, message }`
5. 缓存策略：
   - `GET /api/media` 可加短时 `s-maxage`
   - POST telemetry 禁止缓存

## 6. Cloudflare Pages 部署设计

## 6.1 Pages 项目配置

- Build command: `npm run build:web`
- Build output directory: `dist/web`
- Node version: 与本地一致（建议 20）

## 6.2 Functions 与环境变量

必需变量：

- `UPSTREAM_ORIGIN=https://truvaze.com`
- `UPSTREAM_ANIME_ORIGIN=https://x-ero-anime.com`（如需双源）
- `TELEMETRY_ORIGIN=https://telemetry.x-flow.ccwu.cc`（透传遥测到 Worker）
- `RECOMMEND_ORIGIN=https://telemetry.x-flow.ccwu.cc`（透传推荐到 Worker）

前端变量：

- `VITE_API_BASE=/api`

## 6.3 路由与静态规则

- 自定义域名：`x-flow.ccwu.cc`

- SPA 回退：`/* -> /index.html`（如 web 端使用 history 路由）
- 安全头建议：CSP、`X-Content-Type-Options: nosniff`

## 7. 风险与应对

1. **行为差异风险（GM vs fetch）**  
   通过 Adapter contract 测试（同一输入输出一致）

2. **缓存污染风险**  
   `ApiClient` cache key 保持 `category + perPage + range + channel`

3. **启动策略冲突风险**  
   userscript 和 web bootstrap 分文件，禁止互相引用接管逻辑

4. **线上代理被滥用风险**  
   Functions path 白名单 + method 白名单 + 来源校验（可选）
